# Ansible Configuration — k3s Cluster

Production-grade Ansible automation for bootstrapping and managing a 3-node HA k3s cluster with embedded etcd.

## Architecture Overview

This layer uses a **role-based architecture** with proper separation of concerns:

- **Preflight role**: Read-only validation checks
- **Common role**: Base OS configuration (time sync, swap, sysctl, packages)
- **k3s_server role**: Cluster bootstrap and node joining

- **nfs_client role**: NFS mount configuration

## Quick Start

```bash
# 1. Install collections
ansible-galaxy collection install -r requirements.yml

# 2. First-time bootstrap (runs preflight → common → k3s → nfs)
ansible-playbook bootstrap.yml

# 3. Or run individually
ansible-playbook playbooks/preflight.yml
ansible-playbook playbooks/common.yml
ansible-playbook playbooks/k3s-init.yml
ansible-playbook playbooks/nfs-client.yml
ansible-playbook playbooks/verify.yml
```

The inventory file (`inventory/hosts.ini`) is **auto-generated by Terraform**. Do not edit it manually.

After running `terraform apply` in `../terraform/proxmox/`:

```bash

```

| Group | Purpose |
|-------|---------|
| `k3s_servers` | All 3 k3s nodes |
| `k3s_first_node` | First node (runs `--cluster-init`) |
| `k3s_other_nodes` | Remaining nodes (join existing cluster) |

## Configuration

All configuration is in `group_vars/` (DRY principle):

| File | Purpose |
|------|---------|
| `group_vars/all.yml` | Shared variables (k3s version, NFS settings, paths) |
| `group_vars/k3s_servers.yml` | Cluster-specific (server flags, resource reservations, etcd) |

Key settings:

```yaml
# group_vars/all.yml
k3s_version: "v1.35.3+k3s1"

# NFS with performance tuning
nfs_mount_options: "vers=4.2,hard,intr,timeo=600,retrans=2,rsize=1048576,wsize=1048576,_netdev"
```

## Playbooks

| Playbook | Purpose | Usage |
|----------|---------|-------|
| `site.yml` | **Master playbook** — runs everything in order | `ansible-playbook site.yml` |
| `bootstrap.yml` | **Alias for first-time setup** | `ansible-playbook bootstrap.yml` |
| `playbooks/preflight.yml` | Validation checks (read-only) | `--tags preflight` |
| `playbooks/common.yml` | Base OS configuration | `--tags common` |
| `playbooks/k3s-init.yml` | Bootstrap k3s cluster | `--tags k3s` |
| `playbooks/nfs-client.yml` | Configure NFS mounts | `--tags nfs` |
| `playbooks/verify.yml` | Post-deployment verification | `--tags verify` |

| `playbooks/k3s-etcd-snapshot.yml` | On-demand etcd backup | For Airflow scheduling |
| `playbooks/k3s-uninstall.yml` | **DESTRUCTIVE**: Remove k3s | `-e confirm_uninstall=yes` |
| `playbooks/reboot.yml` | Rolling node reboots | `--check` first |
| `playbooks/kubeconfig-refresh.yml` | Refresh local kubeconfig | After TLS-SAN changes |

## Key Features

### K3s Configuration File

Uses `/etc/rancher/k3s/config.yaml` (k3s v1.19+ canonical approach) instead of `INSTALL_K3S_EXEC` flags:

```yaml
# /etc/rancher/k3s/config.yaml (rendered from template)
tls-san:
  - eve-market.lab.answerisnoh.dev
disable:
  - servicelb
  - traefik
kubelet-arg:
  - "system-reserved=cpu=200m,memory=512Mi,ephemeral-storage=1Gi"
  - "kube-reserved=cpu=200m,memory=512Mi"
  - "eviction-hard=memory.available<300Mi"
```

### Kubelet Resource Reservations

Critical for stability on ~13GB nodes running Airflow (2-3GB) + dlt tasks:
- **System-reserved**: 200m CPU, 512Mi memory (OS daemons)
- **Kube-reserved**: 200m CPU, 512Mi memory (k8s system pods)
- **Eviction-hard**: 300Mi (pods evicted when memory drops below this)

### Rolling Upgrade Safety

- **etcd snapshot** taken on first node before upgrade starts
- **Serial execution**: One node at a time
- **Health gates**: Waits for node Ready and cluster health before proceeding

### NFS Mount Best Practices

- **Version pinned**: NFSv4.2 explicitly (not kernel default)
- **Performance tuned**: 1MB rsize/wsize, hard mount, 60s timeout
- **Pre-check**: Verifies NFS server reachability before mounting
- **Health directory**: Test files go to `.ansible-health/` (not share root)

## Tags

Run specific parts of the deployment:

```bash
# Only validation
ansible-playbook site.yml --tags preflight

# Skip smoke tests
ansible-playbook site.yml --skip-tags smoke

# Only k3s (skip NFS)
ansible-playbook site.yml --tags k3s
```

Available tags:
- `preflight`: Read-only checks
- `common`: Base OS setup
- `bootstrap`: First-time setup tasks
- `k3s`: k3s installation
- `nfs`: NFS configuration
- `install`: Installation tasks
- `config`: Configuration rendering
- `verify`: Verification checks
- `smoke`: Smoke tests
- `summary`: Display summaries

## Upstream Reference

This project uses **custom roles** to demonstrate understanding of the k3s bootstrap sequence. For production environments, consider the official **[k3s-io/k3s-ansible](https://github.com/k3s-io/k3s-ansible)** collection, which is the reference implementation maintained by the k3s team.

**Why custom roles?**
- Visibility into the bootstrap process (pedagogical value)
- Direct control over kubelet resource reservations (critical for homelab RAM budget)
- Custom etcd snapshot and NFS integration logic
- Tailored for this specific 3-node HA embedded-etcd topology

## Troubleshooting

```bash
# Check playbook syntax
ansible-playbook --syntax-check playbooks/k3s-init.yml

# Dry run (check mode)
ansible-playbook --check --diff site.yml

# Verbose output
ansible-playbook -vvv playbooks/k3s-init.yml

# Limit to specific host
ansible-playbook --limit k3s-server-1 playbooks/k3s-init.yml

# Reset fact cache
rm -rf .ansible_cache/
ansible-playbook site.yml
```

## Security Notes

- `k3s-token.txt` and `kubeconfig` files in `inventory/` are **gitignored** (see `.gitignore`)
- Kubeconfig saved to `~/.kube/config` contains cluster admin credentials
- k3s token is transient and should not leave the control node in production
- SSH host key checking is disabled for homelab VMs (`host_key_checking = False`)
